    - Test 3: task_queue 被 put 的任务在 root.after 循环中被执行

  <action>
    在 GUIInterface.__init__ 末尾确保 _process_queue() 已启动（run() 调用前也安全）。实现 GUIThreadBridge 模式（可作为 GUIInterface 方法或嵌套 helper）：

    1. 新增 run_on_main_thread(self, fn, *args, default_on_stop=None, timeout=None) -> Any：
       - 若 threading.current_thread() is threading.main_thread()，直接调用 fn 并返回
       - 否则：创建 threading.Event result_ready、列表 result_holder 和 exception_holder
       - 定义 wrapper：在主线程执行 fn，捕获异常写入 exception_holder，结果写入 result_holder，最后 result_ready.set()
       - 通过 self.task_queue.put(wrapper) 入队（_process_queue 已在主线程 after 循环消费）
       - 工作线程 result_ready.wait()；若 self.should_stop 为 True 返回 default_on_stop
       - 若 exception_holder 非空则 re-raise

    2. 保留现有 task_queue + _process_queue 轮询（100ms），确保每个 task 在主线程执行

    3. 不在此任务中改造 _show_modal_dialog（留给 01-02），但添加 run_on_main_thread 供后续使用

    4. 在 test_gui.py 新增 test_run_on_main_thread_bridge：创建 GUIInterface（可用 mock ConfigManager 或最小 config fixture），在 daemon 线程调用 run_on_main_thread，断言主线程执行成功。CI skip 条件与现有 test_gui 一致。
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker→main (tkinter) | 工作线程不得直接操作 widget；须经 task_queue/root.after |
| 用户关闭窗口 | 主线程销毁 UI 时 worker 可能阻塞等待对话框 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Denial of Service | choice_event.wait 无超时 | mitigate | should_stop 时 set event 并返回哨兵值 |
| T-01-02 | Tampering | worker 直调 widget | mitigate | run_on_main_thread 强制主线程执行 |
| T-01-SC | Tampering | 无新包安装 | accept | 本计划不引入新依赖 |
</threat_model>
